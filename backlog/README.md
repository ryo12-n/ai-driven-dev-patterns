# backlog/

## 用途

具体的な次施策候補の保管場所。1 施策 1 ファイル形式で管理する。

## 起票元

backlog への施策候補は以下の経路で追加される。

| 経路 | 説明 |
|------|------|
| **トリアージ** | triage-manager が inbox・課題管理.csv を走査し、施策化すべきと判断したもの |
| **dev_manager の知見収集** | セッション完了時の report.md で「具体的な次施策候補」と判断されたもの |
| **人間の直接起票** | 人間が施策候補を直接起票する場合 |

## 記載フォーマット

`_template.md` をコピーして新規ファイルを作成する。

```markdown
# <施策名>

## メタ情報

| 項目 | 内容 |
|------|------|
| **優先度** | 高 / 中 / 低 |
| **ステータス** | 候補 |
| **起票日** | YYYY-MM-DD |

## 課題・背景
<!-- 現状の問題点を記述する -->

## 期待効果
<!-- この施策を実施することで得られる効果を記述する -->

## 補足・参考情報
<!-- 任意。関連する議論・参考リンク・技術的な制約など -->
```

## ファイル命名規則

```
backlog/<施策名>.md
```

例: `backlog/spec作成ガイドラインの整備.md`

## ライフサイクル

```
inbox/ ──→ トリアージ走査 ──→ backlog/<施策名>.md ──→ sessions/ で施策開始 ──→ git rm
                                    ↑
課題管理.csv ──→ トリアージ走査 ──┘
                                    ↑
dev_manager report.md ─────────────┘
```

1. トリアージ・知見収集ステップで施策候補を発見 → `backlog/<施策名>.md` を作成
2. 施策として開始する際に、内容を `sessions/` 配下の plan.md に引き継ぐ
3. 施策開始後、backlog のファイルを削除（git 履歴が証跡）

## 優先順位の判断基準

| 優先度 | 基準 |
|--------|------|
| 高 | 開発プロセスの根幹に影響する。早期対応しないとブロッカーになる |
| 中 | 品質向上や効率化に寄与する。計画的に着手すべき |
| 低 | あると便利だが緊急性はない。余裕があるときに着手 |
