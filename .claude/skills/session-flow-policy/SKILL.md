# セッションフロー作成ポリシー

## 適用対象

新しいセッションタイプ（マネージャー＋ワーカー構成）を作成するすべての場面に適用する。

---

## 1. 基本構造: ディスパッチャー → マネージャー → 専門ロール

本リポジトリでは以下の構造でセッションフローを構成する。

```
.claude/skills/dispatcher/SKILL.md    # エントリーポイント（シナリオ分類）
.claude/skills/dispatcher/SKILL.md                  # オーケストレーター（セッション管理）
.claude/agents/<specialist>.md        # 専門ロール（タスク実行）
```

### 1.1 ディスパッチャーをスキルにする理由

- **ユーザーが `/` コマンドで明示的に起動する**ため、`user-invocable: true` のスキルが適切
- スキルの `description` のみが常時読み込まれ、**全文はオンデマンド読み込み**でコンテキストコストが低い

### 1.2 マネージャー（dev_manager）の役割

- セッションディレクトリの作成・管理
- 専門ロールの起動・制御
- 成果物の確認・品質ゲート判定
- ロール間のファイル分離が自然に担保される

### 1.3 専門ロールの役割

- **独立したコンテキストウィンドウ**で動作するため、メインセッションを汚染しない
- dev_manager が Agent ツールで起動・制御する
- ワーカー間のファイル分離が自然に担保される

---

## 2. スキル（SKILL.md）の必須要素

ユーザーが起動するスキルの SKILL.md には以下を含めること。

### 2.1 YAML フロントマター

```yaml
---
name: <skill-name>
description: '<1行の説明>'
user-invocable: true
---
```

### 2.2 必須セクション

1. **概要** — スキルの責務を1〜2文で説明
2. **作業フロー** — 番号付きステップ。ロール起動手順を含む
3. **やること** — スキルの責任範囲
4. **やらないこと** — スコープ外の明示

### 2.3 オーケストレーション関連の必須記載（マネージャーロール）

- **ロールの起動手順**: 起動時に渡す観点（ロール定義パス・スコープ・完了定義）
- **成果物の確認観点**: 品質ゲート判定の基準
- **結果の集約手順**: 全ロール完了後のレポート集約方法
- **担当ファイル**: テーブル形式でファイルと操作権限を明記

---

## 3. ロール定義の必須要素

専門ロール（`.claude/agents/*.md`）には以下を含めること。

### 3.1 必須セクション

`.claude/rules/ (agent-common-workflow.md, agent-restrictions.md, worktree-rules.md)` のフォーマットに準拠し、以下のセクションを含める。

1. **ロール名・概要** — 専門ロールの責務
2. **作業フロー** — 番号付きステップ
3. **やること** — 担当する作業の明示
4. **やらないこと** — スコープ外の明示（マネージャーの責務に属する操作等）
5. **禁止事項** — 明確に禁止される操作

### 3.2 実施者/評価者ペアリングの対称性

実施者と評価者のペアを設ける場合、以下の要素が両方に存在すること。

| 要素 | 実施者に必須 | 評価者に必須 |
|------|------------|------------|
| 壁打ちフェーズ | あり | あり |
| 作業/評価計画 | あり | あり |
| レポート | あり | あり |
| 知見記録セクション | あり | あり |
| 課題起票手段 | あり | あり |

### 3.3 順序制約

- **実施者が完了してから評価者を起動する**（並列起動禁止）
- 評価者は実施者の成果物を直接編集しない（読み取りのみ）

---

## 4. ディスパッチパターン

### 4.1 基本は逐次

ロールは1つずつ（実施 → 評価 → 次のタスク）を基本とする。

### 4.2 並列ディスパッチの条件

並列ディスパッチは以下をすべて満たす場合のみマネージャー判断で許可する。

1. ロール間に入力→出力の依存関係がない
2. 書き込み先ファイルが完全に分離している
3. 並列化による効果がオーバーヘッドを上回る

---

## 5. 既存セッションタイプとの整合性

新しいセッションタイプを作成する際は、既存のセッションタイプとの整合性を確認すること。

| セッションタイプ | エントリーポイント | オーケストレーター | 実行ロール |
|----------------|-----------------|-----------------|-----------|
| 開発セッション | `.claude/skills/dispatcher/` | `.claude/skills/dispatcher/SKILL.md` | `.claude/agents/feature-builder.md`, `.claude/agents/reviewer.md` 等 |
| トリアージセッション | `.claude/skills/triage/` | （スキル内で完結） | — |

### 5.1 新セッションタイプ追加時のチェックリスト

- [ ] スキルまたはロール定義に必須セクション（概要・作業フロー・やること・やらないこと）が含まれている
- [ ] 実施者と評価者のペアを設ける場合、ペアリングの対称性要件を満たしている
- [ ] `docs/` に新セッションタイプのフロー説明を追記した
- [ ] コミットメッセージ規約（`commit-message.md`）に新セッションタイプを追加した（必要な場合）

## 関連ファイル一覧

本スキルファイルの内容を変更した場合、以下のファイルの連動更新が必要になる可能性がある。

| ファイル | 連動更新の内容 |
|---------|-------------|
| `.claude/skills/dispatcher/SKILL.md` | スキルの必須要素の変更がディスパッチャー定義に影響する場合 |
| `.claude/skills/dispatcher/SKILL.md` | オーケストレーション関連の必須記載の変更がマネージャー定義に影響する場合 |
| `.claude/rules/ (agent-common-workflow.md, agent-restrictions.md, worktree-rules.md)` | ロール定義の必須要素の変更が共通ルールに影響する場合 |
| `.claude/rules/commit-message.md` | セッション種別の追加・変更がコミットメッセージ規約に影響する場合 |
