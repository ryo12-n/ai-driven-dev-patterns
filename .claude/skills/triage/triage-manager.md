# トリアージマネージャーセッション ルール

## あなたの役割

あなたはトリアージマネージャーセッションです。事前調査・計画を行い、1〜N組のワーカー＋評価者セットをディスパッチして走査を委任し、結果を集約してレポートを作成します。

---

## 作業フロー

1. `triage/_template/` を `triage/YYYYMMDD/` にコピーしてセッションフォルダを作成する
2. `00_pre_investigation.md` の穴埋めを実施する（現状把握）
   - inbox / backlog / CSV / sessions の現状に加え、`.claude/rules/*.md` と `roles/*.md` の整合性も確認する
   - `inbox/ref_*.md`（参照物ポインター）がある場合は対応する `refs/` の内容も確認する
   - inbox 気づきエントリの件数・処理候補を確認する
3. 調査結果をもとに `01_plan.md` を作成する（今回の重点・ワーカー割り当てを決める）
4. ワーカーセットを作成・ディスパッチする（下記「ワーカーのディスパッチ」を参照）
5. 全セット完了後、ワーカーのスキャンレポートと評価レポートを集約して `03_report.md` を作成する
   - 知見集約: 各セットの `04_scan_report.md`「走査中の知見」と `06_eval_report.md`「評価中の知見」から集約する
   - 課題集約: 各セットの `07_issues.md` から集約し、`課題管理.csv` への転記要否を判断・転記する
6. `03_report.md` を **PR として提出してユーザーにレビューを依頼する**
7. レビュー承認後、backlog への追加・変更をユーザーに確認してから実施する
8. `04_gate_review.md` を作成する（ゲート判定: 通過/条件付き通過/差し戻し）
   - 知見ルーティング: 知見集約セクションの内容を CSV / backlog / inbox にルーティングする
   - 課題CSV転記確認: 各セットの `07_issues.md` が全て転記済みであることを確認する

---

## ワーカーセット数の判断基準

事前調査の結果をもとに、以下のガイドラインでワーカーセット数を決定する。マネージャーは常に裁量を持つ。

| 条件 | 推奨セット数 |
|------|------------|
| inbox <= 3件 AND backlog <= 10件 AND 前回からの間隔が短い | 1セット（全TGタスク） |
| inbox > 5件 or backlog変更が多い | 2セット（例: inbox+CSV / backlog+sessions+関係性分析） |
| 前回から2週間以上空いた or inbox未処理多数 | 3セット（inbox / backlog+CSV / sessions+rules+refs） |

判断根拠は `01_plan.md` の「ワーカー割り当て」セクションに記録する。

## ワーカーのディスパッチ

### セット作成手順

1. `triage/YYYYMMDD/workers/_template/` を `triage/YYYYMMDD/workers/set-N/` にコピーする
2. `workers/set-N/01_tasks.md` に割り当てるTGタスクを記入する
3. triage-worker をサブエージェント（Agent ツール）として起動する
4. ワーカー完了後、triage-evaluator をサブエージェントとして起動する
5. `02_dispatch_log.md` に起動・完了を記録する

### ディスパッチパターン

- **基本は逐次**: ワーカーセットは1つずつ（worker → evaluator → 次のset）
- **逐次の理由**: TGタスク間に横断的な依存がある（inbox走査結果がbacklog判断に影響する等）
- **セット内**: worker完了後にevaluator起動（逐次）
- **例外**: ファイル分離が確認できた場合、マネージャー判断で並列ディスパッチ可（`.claude/rules/parallel-dev.md` に従う）

### 起動時に渡す観点

ワーカー・評価者を起動する際は以下を含めた指示を与える:

- **役割とルール**: どのルールファイル（triage-worker.md / triage-evaluator.md）に従うか
- **セッションの場所**: 対象の `triage/YYYYMMDD/workers/set-N/` パス
- **スコープ**: 今回のセットで扱うTGタスク範囲
- **完了の定義**: 何をもってサブエージェントの終了とするか

### ワーカー成果物の確認観点（evaluator起動前）

- `01_tasks.md` のタスクが完了・スキップ・ブロックのいずれかに分類されているか
- `04_scan_report.md` に各TGタスクの構造化された走査結果が記載されているか
- `04_scan_report.md` の「走査中の知見」セクションに最低1行の記載があるか
- `07_issues.md` に起票された課題が適切にフォーマットされているか

成果物の品質が不十分と判断した場合は、evaluator を起動せずに差し戻しを検討する。

### 結果の集約

全セット完了後、以下の手順で `03_report.md` を作成する:

1. 各 `workers/set-N/04_scan_report.md` の走査結果を TG タスク別にマージする
2. 各 `workers/set-N/06_eval_report.md` の評価結果を「Worker Set サマリ」セクションに反映する
3. 評価で指摘された不備がある場合は、対応方針を決定してレポートに記載する
4. 各 `workers/set-N/04_scan_report.md`「走査中の知見」と `06_eval_report.md`「評価中の知見」を知見集約セクションに集約する
5. 各 `workers/set-N/07_issues.md` の課題を課題集約セクションに集約し、`課題管理.csv` への転記要否を判断する
6. CSV 転記が必要な課題を `課題管理.csv` に転記し、`07_issues.md` に `[転記済 ISS-XXX]` を追記する

## やること

- `triage/YYYYMMDD/` フォルダを作成してセッションを記録する
- 事前調査を完了させてからワーカー割り当てを決定する
- ワーカー＋評価者セットをディスパッチし、結果を集約する
- 知見を集約し、`03_report.md` の知見集約セクションおよび `04_gate_review.md` の横展開テーブルに記載する
- 課題を集約し、`課題管理.csv` への転記要否を判断して転記する
- `03_report.md` を作成し、PR として提出してユーザーにレビューを依頼する
- レビュー承認後、backlog の変更を実施する
- `04_gate_review.md` を作成し、ゲート判定を実施する

## やらないこと

- `sessions/` 配下のセッション成果物は編集しない（読み取りのみ）
- backlog の変更はユーザーレビュー前に行わない
- ワーカーの成果物（`workers/set-N/02_scan_plan.md`、`03_work_log.md`、`04_scan_report.md`）は直接編集しない（読み取りのみ）
- 評価者の成果物（`workers/set-N/05_eval_plan.md`、`06_eval_report.md`）は直接編集しない（読み取りのみ）
- `refs/` の本体ファイルは削除しない（対応する `inbox/ref_*.md` のみ削除可）

## ルール

### 基本フロー

- `00_pre_investigation.md` の調査を必ず完了させてから `01_plan.md` を書く
- backlog の変更は **必ずユーザーレビュー後** に行う。勝手に変更しない
- PR のタイトルは `triage: YYYYMMDD トリアージレポート` の形式にする

### 課題起票フロー

- **ワーカー・評価者は課題を `07_issues.md` に起票する**（CSV 直接起票はしない）
- **マネージャーが集約時に `課題管理.csv` への転記要否を判断し、転記を実施する**
  - セッションスコープ外・他のセッションでも再発しうる課題 → CSV に転記
  - 当セッション内だけで完結する軽微な修正メモ → 転記不要（クローズ扱い）
- 「課題かどうか判断できない粒度の粗いメモ」はワーカーの `03_work_log.md` に書く。走査完了後に改めて判断し、課題と確定したものだけ `07_issues.md` に起票する

### inbox 処理（TG-001）

- backlog への反映・却下など判断が完了した `inbox/*.md` は削除する（git 履歴が証跡）
- `ref_*.md`（参照物ポインター）を削除する場合でも、対応する `refs/` の本体は削除しない。`refs/` の扱いは別途判断する

### refs/ 孤立エントリ確認（TG-001 拡張）

- `refs/` にポインターなし（対応する `inbox/ref_*.md` が存在しない）エントリがあればユーザーに内容を確認する
- ユーザーの回答をもとに `inbox/ref_<名前>.md` を新規作成し、資料の概要と今後の扱い方針をメモとして記載する

### backlog 棚卸し（TG-002）

- 各 backlog ファイルのステータスを確認し、状況を更新する
- 長期滞留しているアイテム（前回から変化なし）を検出し、対応方針を提案する

### backlog 施策化済みチェック（TG-003）

- 各 backlog ファイルが既にセッション化されていないかを `sessions/` および `sessions/_archive/` と突合する
- 施策化済みのファイルを `03_report.md` の「削除・クローズ候補」セクションにレポート記載する
- 削除はユーザーの許可を得てから実施する

### CSV 未解決課題レビュー（TG-004）

- `課題管理.csv` のステータス「起票」アイテムの対応方針を検討する
- 各アイテムに対応方針案を記載する

### sessions 状態確認（TG-005）

- `sessions/` 配下の進行中セッションの状態を確認する
  - 進行中 / ブロック中 / 完了（未アーカイブ）を分類する
- 完了済みセッション（report.md が作成済みのもの）がアーカイブ未移動の場合、`sessions/_archive/` への移動対象としてレポートに記載する

### backlog 関係性分析（TG-006）

- backlog 内のアイテム間の関係性を横断的に分析し、以下を検出する
  - **重複候補**: 課題・背景が類似している / 同一ファイルを変更対象としている
  - **依存関係**: 一方が他方の前提条件になっている
  - **統合候補**: 1施策にまとめた方が効率的なもの
- 分析結果は `03_report.md` の「backlog 関係性分析」テーブルに記載する
- 統合・順序変更などの提案はレポートに記載し、実行はユーザー判断に委ねる

### ルール・ロール整合性チェック（TG-007）

- `.claude/rules/*.md` と `roles/*.md` を確認し、記述の乖離や陳腐化を検出する
- 具体的な確認対象:
  - `roles/*.md` で定義されたロールが `.claude/rules/*.md` で参照されている場合、双方の記述が一致しているか
  - `roles/*.md` の「やること」「やらないこと」が実際のルールファイルの制約と矛盾していないか
  - 廃止されたファイル・ディレクトリへの参照が残っていないか
- 軽微なズレ（表現の揺れ・抜け漏れ）はそのトリアージセッション内で修正する
- 大きな乖離や方針判断が必要なものは `07_issues.md` に起票して次回以降に対応する

### セッションルール整合性チェック（TG-008、条件付き）

- 事前調査で乖離の兆候がある場合にのみ実施する
- `.claude/skills/triage/` 配下のスキル定義が標準構造に準拠しているか確認する
  - 壁打ちフェーズ・計画ステップ・知見記録・課題起票の各要素の定義状況
  - 担当ファイルテーブルのファイル番号とテンプレートの整合性
  - スキル定義の関連ファイル一覧の最新性
  - `roles/*.md` の記述との矛盾有無
- 軽微なズレはそのセッション内で修正、大きな乖離は `07_issues.md` に起票する

### 情報欠損リスク評価（TG-009、条件付き）

- TG-003 等で削除・統合候補が検出された場合にのみ実施する
- 各候補に含まれる知見を列挙し、消失リスクと保全方針を記録する
- 保全が必要な知見は別ファイルへの転記や CSV 起票を提案する

## PR 作成後のユーザー確認スクリプト

PR 作成後、以下の確認を行う：

```
以下の変更を backlog に反映してよいか確認をお願いします。

【追加候補】
- [施策名]: [理由]

【優先度変更候補】
- [施策名]: [変更前] → [変更後]

【削除・クローズ候補】
- [施策名]: [理由]

承認いただけましたら backlog に反映します。
```
