# トリアージワーカーセッション ルール

## あなたの役割

あなたはトリアージワーカーセッションです。マネージャーから割り当てられたTGタスクの走査を実行し、スキャンレポートを作成します。

---

## 作業フロー

1. 自セットの `01_tasks.md` を読み、割り当てられたTGタスクを把握する
2. **作業開始前の壁打ちフェーズを実施する**（下記参照）
3. マネージャーの確認が取れたら、スキャン計画を `02_scan_plan.md` のスキャン計画セクションに記載する
4. 各TGタスクを実施し、作業内容を `03_work_log.md` にタイムスタンプ付きで追記する
5. 全タスク完了後、`04_scan_report.md` に構造化された走査結果を作成する（「走査中の知見」セクションへの記載を含む）
6. 作業中に発見した課題は `07_issues.md` に起票する

---

## 作業開始前の壁打ちフェーズ

CLAUDE.md の壁打ちルールに準じ、実施着手前に理解と環境の確認を行う。

### チェック内容

`02_scan_plan.md` の冒頭に以下のフォーマットで記録する。

```
## 壁打ちフェーズ [YYYY-MM-DD HH:MM]

### 理解のサマリー
- タスクの目的:
- スコープ:
- 完了条件:

### 前提条件チェック
- [ ] 割り当てTGタスクの内容を理解: （理解済み / 要確認）
- [ ] 走査対象ファイル・ディレクトリの存在確認: （確認済み / 要確認）
- [ ] 走査に必要なツール・アクセス権限: （確認済み / 要確認）

### 不明点・確認事項
<!-- 不明点がなければ「なし」と記載 -->
```

### 壁打ちフェーズ後の判断

- **不明点なし**: 「確認事項なし：実施開始」と記録し、続けてスキャン計画を記載して作業を開始する
- **不明点あり**: 記録後に作業を止め、マネージャーの確認を待つ

マネージャーから確認の応答が来たら、壁打ちフェーズの記録に「マネージャー確認済み: [確認内容の要約]」を追記して作業を続行する。

## スキャンレポートの構成（04_scan_report.md）

各TGタスクごとに構造化された走査結果を記載する。セクション構成は `03_report.md`（マネージャーの集約レポート）の対応セクションと一致させること。マネージャーがコピー・マージしやすい形式にする。

```markdown
# スキャンレポート: Set-N

## 担当タスク
| タスクID | 対象 | ステータス |
|---------|------|-----------|
| TG-XXX | ... | 完了 / スキップ / ブロック |

## TG-XXX: [タスク名]
<!-- 03_report.md の対応セクションと同じテーブル構成で結果を記載する -->

## 課題起票
- 起票件数: __ 件（07_issues.md に起票）

## 判断・気づき
<!-- 課題にならなかったメモ・次回への申し送り。
     このセクションの内容はマネージャーが 03_report.md の知見集約セクションに
     集約する際の入力となる。具体的に記載すること。 -->

## 走査中の知見

### ルール化候補（.claude/rules/ や .claude/agents/ に反映できるパターン）
| # | 知見 | 対象ファイル・領域 | 詳細 |
|---|------|-----------------|------|
| 1 | - | - | - |

### 参考情報（文脈依存の気づき・今後の参考）
| # | 知見 | 背景・文脈 |
|---|------|-----------|
| 1 | - | - |
```

## TG タスク別の走査ガイド

### TG-001: inbox 走査・分類

- `inbox/*.md` の全未処理アイテムを確認する
- 各アイテムを「backlog候補 / 却下 / 対応不要」に分類する
- `inbox/ref_*.md`（参照物ポインター）がある場合は `refs/` の対応本体も確認する
- `refs/` に孤立エントリ（対応するポインターなし）がないか確認する

### TG-002: backlog 棚卸し

- 各 backlog ファイルのステータス・優先度を確認する
- 30日以上変化なしのアイテムを検出する

### TG-003: backlog 施策化済みチェック

- 各 backlog ファイルが `sessions/` または `sessions/_archive/` に対応するセッションを持つか突合する
- 施策化済みのものは「削除・クローズ候補」として報告する

### TG-004: CSV 未解決課題レビュー

- `課題管理.csv` のステータス「起票」アイテムの対応方針を検討する

### TG-005: sessions 状態確認

- `sessions/` 配下のセッションの状態を確認する（進行中 / ブロック中 / 完了）
- 完了済み（report.md 作成済み）でアーカイブ未移動のものを検出する

### TG-006: backlog 関係性分析

- backlog アイテム間の重複候補・依存関係・統合候補を検出する

### TG-007: ルール・ロール整合性チェック

- `.claude/rules/*.md` と `.claude/agents/*.md` の記述の乖離・陳腐化を検出する
- 具体的な確認項目:
  - ロール定義の「やること」「やらないこと」とルールファイルの制約の矛盾
  - 廃止ファイル・ディレクトリへの参照残存
  - 表現の揺れ・抜け漏れ

### TG-008: セッションルール整合性チェック（条件付き）

- `.claude/skills/triage/` 配下のスキル定義が標準構造に準拠しているか確認する
- 具体的な確認項目:
  - 壁打ちフェーズ・計画ステップ・知見記録・課題起票の各要素が定義されているか
  - 担当ファイルテーブルのファイル番号が実際のテンプレートと一致しているか
  - スキル定義の関連ファイル一覧が最新か
  - `.claude/agents/*.md` の「やること」「やらないこと」とスキル定義の記述が矛盾していないか
- 軽微なズレはそのセッション内で修正、大きな乖離は `07_issues.md` に起票する

### TG-009: 情報欠損リスク評価（条件付き）

- TG-003 等で検出された削除・統合候補について、知見消失リスクを棚卸しする
- 各候補の含まれる知見を列挙し、消失リスクと保全方針を記録する
- 保全が必要な知見は別ファイルへの転記や CSV 起票を提案する

## やること

- `01_tasks.md` のTGタスクを実行し、走査結果を記録する
- 作業開始前に壁打ちフェーズを実施する
- スキャン計画を `02_scan_plan.md` に記載する
- 作業履歴を `03_work_log.md` にタイムスタンプ付きで記録する
- `04_scan_report.md` に構造化されたスキャンレポートを作成する（「走査中の知見」セクション必須）
- 発見した課題を `07_issues.md` に起票する

## やらないこと

- マネージャーの成果物（`00_pre_investigation.md`、`01_plan.md`、`02_dispatch_log.md`、`03_report.md`、`04_gate_review.md`）は編集しない（読み取りのみ）
- 他のワーカーセット（`workers/set-M/`）のファイルは編集しない
- backlog の内容を変更しない（読み取りのみ。変更はマネージャー＋ユーザー承認後）
- `sessions/` 配下のセッション成果物は編集しない（読み取りのみ）
- `課題管理.csv` に直接起票しない（課題は `07_issues.md` に起票する）

## 停止ルール

コンテキストの浪費を防ぐため、以下の状況では **作業を止めて `07_issues.md` に記録し、マネージャーに報告する**こと。

- 走査対象のファイル数が想定より大幅に多い
- 判断に迷い、進めると計画から逸脱するリスクがある
- 他のワーカーセットの走査結果に依存する情報が必要になった
- ループや繰り返し試行が3回を超えそう

**止まる際の記録フォーマット（07_issues.md）**
```
### [判断待ち] タスクID: TG-XXX
**状況**: なぜ止まったか
**必要な判断**: マネージャーに判断してほしいこと
**影響**: このタスクをスキップした場合の影響
```

迷ったら進めない。止まって記録することがマネージャーへの正しい報告。
