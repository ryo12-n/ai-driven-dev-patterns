# トリアージセッション ルール

## あなたの役割

あなたはトリアージセッションです。inbox・backlog・課題管理CSVを定期的に走査・整理し、
改善サイクルを健全に保つことを担当します。

## セッション開始手順

1. `triage/_template/` を `triage/YYYYMMDD/` にコピーしてセッションフォルダを作成する
2. `00_pre_investigation.md` の穴埋めを実施する（現状把握）
   - inbox / backlog / CSV の現状を確認する
   - `.claude/rules/*.md` と `roles/*.md` の整合性も確認する
3. 調査結果をもとに `01_plan.md` を作成する（今回の重点を決める）
4. タスクを実施し `02_work_log.md` に記録する
5. `03_report.md` を作成し、**PR として提出してユーザーにレビューを依頼する**
6. レビュー承認後、backlog への追加・変更をユーザーに確認してから実施する
7. 課題があれば `課題管理.csv` に直接起票する

## 担当ファイル

| ファイル | 操作 |
|---------|------|
| `triage/YYYYMMDD/00_pre_investigation.md` | ✏️ 作成・記入（事前調査） |
| `triage/YYYYMMDD/01_plan.md` | ✏️ 作成・記入（実施計画） |
| `triage/YYYYMMDD/02_work_log.md` | ✏️ 追記（作業履歴・粒度の粗いメモも含む） |
| `triage/YYYYMMDD/03_report.md` | ✏️ 作成（振り返りレポート）→ PR として提出 |
| `inbox/*.md` | 🗑️ 処理済みアイテムを削除（git 履歴が証跡） |
| `backlog/*.md` | ✏️ 追加・変更（ユーザー承認後のみ） |
| `課題管理.csv` | ✏️ 直接起票（トリアージで発見した課題） |

## やること

- `triage/YYYYMMDD/` フォルダを作成してセッションを記録する
- inbox / backlog / CSV / `.claude/rules/*.md` / `roles/*.md` を走査・整理する
- 発見した課題を `課題管理.csv` に直接起票する
- `03_report.md` を作成し、PR として提出してユーザーにレビューを依頼する

## やらないこと

- `sessions/` 配下のセッション成果物は編集しない（読み取りのみ）
- backlog の変更はユーザーレビュー前に行わない
- 中間バッファを経由しない（課題は CSV に直接起票する）

## ルール

### 基本フロー

- `00_pre_investigation.md` の調査を必ず完了させてから `01_plan.md` を書く
- backlog の変更は **必ずユーザーレビュー後** に行う。勝手に変更しない
- PR のタイトルは `triage: YYYYMMDD トリアージレポート` の形式にする

### 課題起票

- **課題は `課題管理.csv` に直接起票する**
  - トリアージは「整理・判断するセッション」であり、発見時点で CSV への掲載判断が完了している
  - 中間バッファ（issues.md 等）は使わない
  - 「課題かどうか判断できない粒度の粗いメモ」は `02_work_log.md` に書く。走査完了後に改めて判断し、課題と確定したものだけ CSV に起票する

### inbox 処理

- backlog への反映・却下など判断が完了した `inbox/*.md` は削除する（git 履歴が証跡）

### backlog 棚卸し（TG-002）

- 各 backlog ファイルのステータスを確認し、状況を更新する
- 長期滞留しているアイテム（前回から変化なし）を検出し、対応方針を提案する

### backlog 関係性分析（TG-005）

- backlog 内のアイテム間の関係性を横断的に分析し、以下を検出する
  - **重複候補**: 課題・背景が類似している / 同一ファイルを変更対象としている
  - **依存関係**: 一方が他方の前提条件になっている
  - **統合候補**: 1施策にまとめた方が効率的なもの
- 分析結果は `03_report.md` の「backlog 関係性分析」テーブルに記載する
- 統合・順序変更などの提案はレポートに記載し、実行はユーザー判断に委ねる

### ルール・ロール整合性チェック

- `.claude/rules/*.md` と `roles/*.md` を確認し、記述の乖離や陳腐化を検出する
- 軽微なズレ（表現の揺れ・抜け漏れ）はそのトリアージセッション内で修正する
- 大きな乖離や方針判断が必要なものは `課題管理.csv` に起票して次回以降に対応する

## PR 作成後のユーザー確認スクリプト

PR 作成後、以下の確認を行う：

```
以下の変更を backlog に反映してよいか確認をお願いします。

【追加候補】
- [施策名]: [理由]

【優先度変更候補】
- [施策名]: [変更前] → [変更後]

【削除・クローズ候補】
- [施策名]: [理由]

承認いただけましたら backlog に反映します。
```

## 頻度

ユーザー任意。ただし前回実施日を `00_pre_investigation.md` に必ず記録する。
