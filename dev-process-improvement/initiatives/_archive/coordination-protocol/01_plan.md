# 実施計画: 協調プロトコル整備（commit/lock/test）

## フェーズ構成

| フェーズ | 内容 | 期間 | 成功基準 |
|---------|------|------|----------|
| 1 | 調査・プロトコル設計・ガイドライン文書作成 | 1セッション | 3つのプロトコル（commit/lock/test）が定義されたガイドライン文書が作成されている |
| 2 | 評価・レビュー | 1セッション | 評価レポートでプロトコルの実用性・整合性が確認されている |

## スケジュール

1. **前提**: git worktree 施策（`initiatives/git-worktree-standardization/`）のフェーズ1 が完了していること
2. フェーズ1: L2-worker が参考 PoC の分析・プロトコル設計・ガイドライン文書を作成
3. フェーズ2: L2-evaluator がプロトコルの品質・実用性を評価
4. L1 がゲート判定を実施

## 成功基準（全体）

1. commit プロトコル（タイミング・メッセージ規約・コンフリクト解消手順）が定義されている
2. lock プロトコル（排他制御の仕組み・対象リソースの粒度・デッドロック回避策）が定義されている
3. test プロトコル（テスト実行の順序・範囲・失敗時の振る舞い）が定義されている
4. 参考 PoC の設計が分析され、本リポジトリ向けにカスタマイズされている
5. git worktree 施策のガイドラインとの整合が確認されている

## リソース・前提条件

- git worktree 施策の成果物（`docs/git-worktree-guideline.md`）が利用可能であること
- 参考 PoC（`refs/ai-driven-development-poc/`）のプロトコルドキュメントが参照可能であること
- 現在の L1/L2 ワークフロー（`docs/workflow.md`、`.claude/rules/`）が参照可能であること

## リスクと対策

| リスク | 発生確率 | 影響 | 対策 |
|--------|---------|------|------|
| worktree 施策の成果が想定と異なり、プロトコル設計に影響する | 中 | 中 | worktree 施策のガイドラインを読んでからプロトコル設計に着手する |
| プロトコルが理論的になりすぎて実用性が低い | 低 | 中 | 具体的なシナリオ（順次実行 / 並列実行）ごとにプロトコル適用例を記載する |
| 3つのプロトコルの相互依存が複雑になりスコープが膨らむ | 低 | 低 | 各プロトコルを独立して定義し、相互の接続点のみを明記する |
