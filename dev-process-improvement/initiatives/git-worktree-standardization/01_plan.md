# 実施計画: git worktree によるClaude並列開発の標準化検討

## フェーズ構成

| フェーズ | 内容 | 期間 | 成功基準 |
|---------|------|------|----------|
| 1 | 調査・ガイドライン文書作成 | 1セッション | ガイドライン文書が作成され、判断基準が明文化されている |
| 2 | 評価・レビュー | 1セッション | 評価レポートで文書の実用性・網羅性が確認されている |

## スケジュール

1. フェーズ1: L2-worker が調査・ガイドライン文書を作成
2. フェーズ2: L2-evaluator がガイドラインの品質・実用性を評価
3. L1 がゲート判定を実施

## 成功基準（全体）

1. Claude Code の `isolation: "worktree"` 機能の仕様・制約が調査結果としてまとめられている
2. worktree を使う場面と使わない場面の判断基準が明確に文書化されている
3. 参考 PoC のロックベース方式との比較分析が含まれている
4. 現在の L1/L2 ワークフローとの関係が整理されている
5. 協調プロトコル施策との接続点が明記されている

## リソース・前提条件

- Claude Code の公式ドキュメント・Web 情報から worktree 機能の仕様を調査可能であること
- 参考 PoC（`refs/ai-driven-development-poc/`）のドキュメントが参照可能であること
- 現在の L1/L2 ワークフロー（`docs/workflow.md`、`.claude/rules/`）が参照可能であること

## リスクと対策

| リスク | 発生確率 | 影響 | 対策 |
|--------|---------|------|------|
| worktree 機能の公式情報が限定的で調査が不十分になる | 中 | 中 | Web 検索で最新情報を補完。不明な点は「要検証」として明記する |
| ガイドラインが抽象的になりすぎて実用性が低い | 低 | 中 | 具体的なユースケースと判断フローチャートを含める |
| 調査範囲が広がりすぎてスコープを超える | 低 | 低 | タスク定義でスコープを明確にし、L2 の停止ルールで逸脱を防ぐ |
