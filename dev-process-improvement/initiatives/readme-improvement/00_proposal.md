# 改善施策提案: README 改善（起動方法整理・プロンプト方式変更）

## 背景・課題

`dev-process-improvement/README.md` のセッション起動方法に以下の問題がある。

### 課題1: 起動方法の選択肢が多く、整理されていない

起動方法が「方法1〜3」の3択で記載されており、どれを使うべきかが不明確な状態。

- **方法1**: Claude Code Agent Teams（推奨・L1がTaskツールでL2を管理）
- **方法2**: 手動で3セッション起動
- **方法3**: Claude Projects（Web版）

「推奨」と書かれているだけで方法2・3が現役かどうか不明であり、セッション開始時に毎回判断コストが発生している。

### 課題2: 起動プロンプトが長文で、何を渡すべきかが不明確

現状のプロンプトは100字超の長文で、「どのルールに従うか」「どの施策を扱うか」が混在している。
テンプレート（`templates/l1-prompts.md`）も同様に冗長で、コピペして使いにくい状態。

## 目標

1. 起動方法を**マネージャー・ワーカー方式**に一本化し、手動起動・Web版の記述は削除する
2. 起動プロンプトを「ルールファイル指定 + 施策の指定」だけの短い宣言型に変更する
3. `l1-manager.md` にプロンプト入力パターンの解釈ロジックを追加する
4. `templates/l1-prompts.md` を廃止する

## 変更後のプロンプト形式（確定）

```
# 新施策開始パターン
.claude/rules/l1-manager.md に従う
backlog: README改善（起動方法整理・プロンプト方式変更）

# 既存施策継続パターン
.claude/rules/l1-manager.md に従う
施策: initiatives/readme-improvement/
```

## l1-manager.md のフォルダ指定時の自動判断ロジック（確定）

| フォルダの状態 | 動作 |
|-------------|------|
| `00_proposal.md`〜`02_tasks.md` が存在、`03_work_log.md` 以降がない | L2-worker 起動モード |
| `04_work_report.md`・`06_eval_report.md` が揃っている | ゲート判定モード |
| `00_proposal.md` が存在しない | 提案・計画作成から開始 |

## スコープ

### やること

- `README.md` の起動方法セクションをマネージャー・ワーカー方式のみに書き直す（手動起動・Web版の記述を削除）
- `README.md` の起動プロンプト例を宣言型の2パターンに変更。L2起動プロンプトを削除する
- `.claude/rules/l1-manager.md` に入力パターンの解釈とフォルダ状態の自動判断ロジックを追記する
- `templates/l1-prompts.md` を削除する
- `templates/l2-prompts.md` を確認のうえ、不要なら削除する
- `docs/workflow.md` に起動方法の記述がある場合は合わせて更新する

### やらないこと

- `.claude/rules/*.md` の既存ルール内容の変更（追記のみ）
- Claude Code の設定ファイル（`settings.json`）の変更
- 起動スクリプト（bash ラッパー等）の新規作成

## 期待される効果

- 起動方法の選択に迷わない（マネージャー・ワーカー方式の1択）
- プロンプトが2行になり、施策の切り替えがパス/施策名の差し替えだけで済む
- フォルダ状態で動作が自動決定されるため、「次に何をすればいいか」を L1 が自律判断できる

## リスク

| リスク | 影響度 | 対策 |
|--------|--------|------|
| 手動起動・Web版を使っていた場合に記述が失われる | 低 | git 履歴が証跡として残るため対応不要 |
| フォルダ状態の自動判断で誤ったモードに入る | 中 | 判断ロジックを l1-manager.md に明示し、L1 が判断根拠を `03_work_log.md` に記録するルールを追加する |
