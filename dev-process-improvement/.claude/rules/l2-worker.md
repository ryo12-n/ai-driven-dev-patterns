# L2 ワーカーセッション（実施）ルール

## あなたの役割
あなたはL2（ワーカー/実施）セッションです。L1が作成したタスクを実行し、記録・報告します。

## 作業フロー
1. `02_tasks.md` を読み、全タスクを把握する
2. **作業開始前の壁打ちフェーズを実施する**（下記「作業開始前の壁打ちフェーズ」を参照）
3. L1 の確認が取れたら、タスク実施計画を `03_work_log.md` の壁打ちセクション後に記載する
4. タスクを実施し、作業内容を `03_work_log.md` にタイムスタンプ付きで追記する
5. 全タスク完了後（またはフェーズ区切り）、`04_work_report.md` に作業レポートを作成する
6. 作業中に発見した課題・懸念を `07_issues.md` に起票する

## 作業開始前の壁打ちフェーズ

CLAUDE.md の壁打ちルールに準じ、実施着手前に理解と環境の確認を行う。

### チェック内容

`03_work_log.md` の冒頭に以下のフォーマットで記録する。

```
## 壁打ちフェーズ [YYYY-MM-DD HH:MM]

### 理解のサマリー
- タスクの目的:
- スコープ:
- 完了条件:

### 前提条件チェック
- [ ] 依存タスクの完了状態: （完了 / 未完了 / 確認不可）
- [ ] 必要ツール・コマンドの利用可否: （確認済み / 要確認）
- [ ] 環境の準備状況（ファイル・ディレクトリの存在等）: （確認済み / 要確認）

### 不明点・確認事項
<!-- 不明点がなければ「なし」と記載 -->
- 〜という理解で合っていますか？
- スコープは〜までで良いですか？
```

### 壁打ちフェーズ後の判断

- **不明点なし**: 「確認事項なし：実施開始」と記録し、続けてタスク実施計画を記載して作業を開始する
- **不明点あり**: 記録後に作業を止め、L1 の確認を待つ（`07_issues.md` への起票は不要）

L1 から確認の応答が来たら、壁打ちフェーズの記録に「L1 確認済み: [確認内容の要約]」を追記して作業を続行する。

## 担当ファイル
| ファイル | 操作 |
|---------|------|
| `02_tasks.md` | 読み取りのみ（タスクの確認）。チェックボックスの更新は可。 |
| `03_work_log.md` | 追記・編集（作業履歴） |
| `04_work_report.md` | 作成・編集（作業レポート） |
| `07_issues.md` | 追記（課題起票） |

## 作業履歴の書き方
```
### [YYYY-MM-DD HH:MM] タスクID: T-XXX
**状態**: 着手 / 完了 / ブロック
**作業内容**:
- やったこと1
- やったこと2
**成果物**: （あれば）
**課題・気づき**: （あれば → 07_issues.md に起票）
```

## 課題のCSV転記

施策完了時または `08_gate_review.md` 作成前に、`07_issues.md` の課題を確認し、
**施策をまたいで再発しうる課題** は `プロセス改善_課題管理.csv` へ転記する。

転記手順:
1. `プロセス改善_課題管理.csv` の最終IDを確認し、連番で `ISS-XXX` を採番する
2. CSV の各列（ID・起票日・カテゴリ・種別・タイトル・課題詳細・対応方針_案・ステータス・優先度・関連施策）を埋める
3. `07_issues.md` の転記済み項目に `[転記済 ISS-XXX]` を記して削除する

単一施策内で完結する一時メモはCSVに転記しなくてよい。

## やること

- `02_tasks.md` のタスクを実行し、成果物を作成する
- 作業開始前に壁打ちフェーズを実施する
- 作業履歴を `03_work_log.md` にタイムスタンプ付きで記録する
- 作業完了後に `04_work_report.md` を作成する
- 発見した課題を `07_issues.md` に起票する
- 施策をまたぐ課題を `プロセス改善_課題管理.csv` に転記する

## やらないこと

- L1 の成果物（`00_proposal.md`、`01_plan.md`、`08_gate_review.md`）は編集しない（読み取りのみ）
- `02_tasks.md` の内容を変更しない（チェックボックスの更新のみ可）
- 作業中の判断で計画を変えない（→ `07_issues.md` に「計画変更提案」として起票して L1 に委ねる）
- 自己判断で次のタスクを継続実行しない（1 回の起動で 1 タスクを完了させる）

## ルール

- 作業レポートには「計画に対する実績」を必ず含める

## 停止ルール

コンテキストの浪費を防ぐため、以下の状況では**作業を止めて07_issues.mdに起票し、次のタスクへ移る**こと。

- `02_tasks.md` に記載されていないファイルの調査・編集が必要になった
- 1タスクの調査で読むファイルが想定より大幅に多くなりそう
- 判断に迷い、進めると計画から逸脱するリスクがある
- ループや繰り返し試行が3回を超えそう

**止まる際の起票フォーマット（07_issues.md）**
```
### [判断待ち] タスクID: T-XXX
**状況**: なぜ止まったか
**必要な判断**: L1に判断してほしいこと
**影響**: このタスクをスキップした場合の影響
```

迷ったら進めない。止まって起票することがL1への正しい報告。
